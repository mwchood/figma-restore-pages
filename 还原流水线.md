# Figma 页面还原流水线

这是本项目唯一有效的流水线入口。原则、优先级和禁止事项见 [还原规则.md](./还原规则.md)。

## 1. 任务类型

用户提供 Figma 节点链接，并指定或默认执行一种任务：

- `还原并发布`：读取、实现、四层审计、加入菜单、部署和线上验证。
- `仅本地还原`：完成实现和四层审计，不发布。
- `先找原因，不修改`：只读取数据、浏览器状态和代码，输出根因。
- `修复并发布`：修复现有页面，补充回归检查并同步线上。

素材、字体、权限或视觉工具缺失时必须先报告，不允许自动降级。

## 2. 标准流程

### 阶段 0：读取与回归保护

1. 读取 `还原规则.md` 和本文件。
2. 检查工作区状态，不覆盖用户已有改动。
3. 确认本次涉及的历史同类问题和已有专项审计。
4. 文档或脚本重构时，先列出将被迁移的检查项，禁止无落点归档。

### 阶段 1：读取 Figma

1. 对目标 Frame 调用 design context。
2. 获取 metadata，确认一级 Frame、父子层级、尺寸和坐标。
3. 获取同一节点的 Figma 截图作为视觉基准。
4. 大型页面按子节点拆分读取，禁止只依赖截图推断。

### 阶段 2：素材本地化

1. 识别图标、图片、背景、渐变、纹理、遮罩和复杂矢量的真实节点。
2. 立即下载临时素材到 `assets/`。
3. 检查文件真实格式、natural size、透明背景和包装层。
4. 删除未引用素材，HTML 中不得残留临时 MCP URL。

### 阶段 3：实现

1. 按项目现有 HTML/CSS 方式实现。
2. 保留根 Frame 和关键节点的 `data-figma-id`。
3. 先换算父子坐标，再实现约束、居中、基线、裁切和层叠关系。
4. 替代字体加载后重新校对文本框，不照搬原字体坐标。

### 阶段 4：建立 manifest

每个页面必须在 `restore-manifests/` 建立清单，至少包含：

- `page`、`fileKey`、`nodeId`。
- `sourceFrame`、`renderScale`、`viewport`。
- `frameSelector` 和 `stylesheet`。
- `visualBaseline` 和 `visualBaselineSha256`：固定的 Figma 基准及防篡改哈希。
- `visualDiff`：原始色差、门禁色差和整页最大差异比例。
- 关键 `texts`。
- `assets` 及 SVG `naturalSize`。
- `cssChecks`：关键静态坐标与尺寸。
- `browserChecks`：真实浏览器中的几何关系和文字适配。
- `visualRegions`：必须局部截图检查的高风险区域。
- `audits`：需要执行的专项审计脚本。

参考：[restore-manifests/untitled-schedule.json](./restore-manifests/untitled-schedule.json)

### 阶段 5：四层审计

1. 运行统一流水线和专项审计。
2. 静态检查失败时先修复来源、素材、CSS 或 manifest。
3. 浏览器检查失败时以实际 DOM bounds 为准修复。
4. 查看整页截图及所有 `visualRegions` 局部截图。
5. 与 Figma 基准逐项比较，不允许只看缩略图。
6. 修复后重新运行完整流水线，不能只重跑失败片段。

### 阶段 6：回归验证

用户指出过的问题必须执行一次“先失败、后通过”验证：

1. 在未修复状态下证明新增检查能发现问题。
2. 修复页面。
3. 证明同一检查通过。
4. 检查相邻页面是否存在同类问题。

无法自动化的问题必须加入 `visualRegions`，并在交付时明确人工检查结论。

### 阶段 7：导航与发布

1. 加入 `index.html` 和 `figma-restore-pages.html`。
2. 更新统一导航审计。
3. 提交并推送 GitHub。
4. 等待 GitHub Pages 部署成功。
5. 验证线上导航、HTML 和关键素材返回 200。

### 阶段 8：交付说明

交付时分别说明：

- 来源与素材审计结果。
- 运行时几何和文本审计结果。
- 已检查的局部视觉区域。
- 字体或环境造成的已知限制。
- 线上地址与提交记录。

## 3. Manifest 浏览器检查

`browserChecks` 用于检查 CSS 源码无法证明的实际渲染关系。

### 元素对齐

```json
{
  "type": "align",
  "selector": ".selected-time",
  "relativeTo": ".selection-background",
  "axis": "centerY",
  "tolerance": 1
}
```

支持的轴：`left`、`right`、`top`、`bottom`、`centerX`、`centerY`。

### 实际 bounds

```json
{
  "type": "bounds",
  "selector": ".primary-button",
  "expected": { "x": 20, "y": 742, "width": 335, "height": 50 },
  "tolerance": 0.5
}
```

### 文字适配

```json
{
  "type": "textFit",
  "selector": ".card-title"
}
```

`textFit` 检查元素自身是否出现横向或纵向裁切。多行文字还必须通过局部视觉检查确认换行位置。

## 4. 强制视觉差异证据

`visualRegions` 为人工视觉验收清单：

```json
{
  "name": "selected-time-row",
  "selector": ".time-picker-selection",
  "padding": 8,
  "maxMismatchRatio": 0.04
}
```

流水线对整页和每个区域都必须输出：

```text
_figma_tmp/pipeline/<page>-visual-diff/
├── <region>--figma.png
├── <region>--browser.png
├── <region>--diff.png
├── report.json
└── index.html
```

- `figma.png`：固定 Figma 基准裁切。
- `browser.png`：同坐标的真实浏览器裁切。
- `diff.png`：超过感知色差阈值的像素标红。
- `report.json`：记录区域、原始差异、感知差异、差异包围盒和门禁结论。
- `index.html`：把每个区域的三张图并排展示，供交付前逐项查看。

每个区域需要检查：

- 位置、尺寸和裁切。
- 图标方向、清晰度和意外底色。
- 字号、字重、行高、基线和视觉中心。
- 渐变、纹理、光效和透明度。

整页截图不能替代局部截图。

### 双差异指标

- `rawMismatchRatio` 使用较低色差阈值，完整暴露抗锯齿、字形和细微颜色变化。
- `mismatchRatio` 使用按区域配置的感知色差阈值，超过 `maxMismatchRatio` 时阻止交付。
- 几何 bounds、中心、基线和文字溢出由独立浏览器检查负责，不能用较高像素容差绕过。
- 阈值必须根据区域类型设置；文字、照片和渐变可以高于纯色图标，但不能根据一次失败随意提高到刚好通过。

### 基准更新流程

1. 确认 Figma 节点确实发生变化。
2. 重新获取精确 Frame 截图到 `visual-baselines/`。
3. 更新 manifest 中的 SHA-256。
4. 重新运行整页和所有局部差异。
5. 人工检查三张证据图后提交。

禁止流水线自动接受新基准。

## 5. 运行命令

```powershell
node scripts/restore-pipeline.cjs restore-manifests/untitled-schedule.json
```

Codex 桌面环境中若系统 Node.js 不含 Playwright，必须使用工作区依赖提供的 Node.js；不得因此跳过浏览器验收。

## 6. 自动检查范围

统一流水线必须覆盖：

- Figma 元数据、Frame 尺寸、渲染比例和关键文本。
- 临时 MCP URL、截图裁切、内联替代 SVG 和降级标记。
- 本地素材存在性、引用、真实格式和 SVG natural size。
- 常见灰底、白底和错误包装层。
- 关键 CSS 坐标和尺寸。
- 专项审计脚本。
- Chrome/Edge 中的图片加载、根画布起点、尺寸和页面溢出。
- manifest 声明的 bounds、对齐关系和文字适配。
- Figma 基准尺寸、SHA-256 和页面来源。
- 整页及局部区域的 Figma 图、浏览器图、差异图和 JSON 报告。
- 超过区域差异阈值时自动失败。
- 统一导航和文档入口完整性。

## 7. 自动检查不能替代

- 选择正确的 Figma 语义节点。
- 判断导出素材中的包装层是否属于设计。
- 替代字体的字形观感和光学补偿。
- 复杂图形端点、渐变、阴影、光效和抗锯齿。
- Figma 与浏览器局部截图的最终人工判断。

这些内容不能被省略，只能转入明确的 `visualRegions` 验收。

## 8. 发布门槛

只有同时满足以下条件才可描述为“发布完成”：

- 真实 Figma 数据和素材已读取并本地化。
- manifest 完整。
- 静态、专项和浏览器运行时审计通过。
- 所有强制局部区域已完成人工对比。
- 用户指出过的问题已有回归检查或明确人工检查项。
- 已说明字体与环境限制。
- 线上导航、页面和关键素材验证成功。

任何一项未完成，必须明确报告当前停留在哪一层，不能笼统声称审计或还原已经完成。
