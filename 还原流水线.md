# Figma 页面还原流水线

这是本项目唯一有效的流水线入口。规则和禁止事项见 [还原规则.md](./还原规则.md)。

## 我们如何配合

用户只需提供目标 Figma 节点链接，并说明以下一种任务类型：

- `还原并发布`：完整实现、验收、加入菜单并部署。
- `仅本地还原`：完成本地实现和验收，暂不发布。
- `先找原因，不修改`：只读取数据并诊断。
- `修复并发布`：修复现有页面、验收并同步线上。

Codex 负责读取规则、获取 Figma 数据和真实素材、实现页面、建立 manifest、运行流水线、人工视觉对比和线上验证。素材或工具缺失时必须先向用户说明，不允许自动降级。

## 新页面流程

1. 读取 `还原规则.md` 和本文件。
2. 通过 Figma MCP 获取 design context、metadata 和截图。
3. 下载真实素材并检查 SVG 内容。
4. 实现页面并保留 Figma 节点追溯信息。
5. 在 `restore-manifests/` 新建页面清单。
6. 运行统一流水线。
7. 查看 `_figma_tmp/pipeline/` 中的浏览器截图，与 Figma 做局部对比。
8. 修正差异并重新运行流水线。
9. 加入统一菜单，提交并部署。
10. 验证线上页面和关键素材返回 200。

## 页面清单

每个 manifest 至少包含：

- 页面文件名、Figma file key 和 node id。
- 源 Frame 尺寸、渲染比例和浏览器视口。
- 根画布选择器。
- 关键文字。
- 本地素材路径和 SVG natural size。
- 关键 CSS 坐标与尺寸。
- 需要运行的专项审计脚本。

参考：[restore-manifests/typhur-test-schedule.json](./restore-manifests/typhur-test-schedule.json)

## 运行命令

```powershell
node scripts/restore-pipeline.cjs restore-manifests/typhur-test-schedule.json
```

在 Codex 桌面环境中，如果系统没有 `node` 命令，使用工作区提供的 Node.js 运行同一个脚本。

## 自动检查内容

- Figma file key、node id、Frame 尺寸和渲染比例。
- 临时 MCP URL、截图裁切、内联手绘 SVG 和降级标记。
- 本地素材存在性、页面引用和 SVG natural size。
- 常见灰色导出背景。
- 关键 CSS 坐标和尺寸。
- 页面对应的专项审计。
- 真实 Chrome/Edge 中的图片加载、画布起点、渲染尺寸和页面溢出。
- 目标视口截图尺寸。
- 根目录只能保留 `README.md`、`还原规则.md` 和 `还原流水线.md`，其他 Markdown 必须归档。

截图输出到 `_figma_tmp/pipeline/`，不会提交到 Git。

## 自动检查不能替代

- 选择正确的 Figma 语义节点。
- 识别素材中不属于设计内容的包装层。
- 判断字体观感和替代字体补偿。
- 检查复杂图表、圆环端点、渐变、光效和组合图层。
- 最终浏览器与 Figma 的人工视觉对比。

## 交付标准

只有同时满足以下条件才可以发布：

- 真实 Figma 数据已读取。
- 关键真实素材已本地化。
- manifest 完整。
- 流水线和专项审计通过。
- 高风险区域已完成人工截图对比。
- 已说明无法达到 1:1 的字体或环境限制。
- 线上页面和关键素材验证成功。
